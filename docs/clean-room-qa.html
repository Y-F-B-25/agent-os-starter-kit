Clean-Room Rules

The QA run must behave like a stranger found the repo, picked a co-pilot, and asked it to set up an agent operating system from scratch.

Excluded

Private Command Center

The starter kit does not ship the maintainer's private Command Center, group chat, private routes, inboxes, or production telemetry.

Input

Fresh Repo Copy

Use a temporary folder and only the files in this repo. Do not rely on prior agent memory or previous setup notes.

Input

Blank Vault

Start with a new vault folder and let the setup co-pilot create the structure, index, progress file, and handoff loop.

Constraint

No Inside Knowledge

The tester answers like a first-time user. The co-pilot can read repo docs, but cannot use private operational context.

First-Time User Flow

Each lane should move through the same core proof path, even if the interface feels different.

01 Open START_HERE.md User pastes the boot prompt into the chosen setup co-pilot.
02 Run diagnostic Co-pilot asks goals, tools, memory state, risk constraints, and success test.
03 Pick lane Co-pilot recommends Cowork, Claude Code, Codex, or blended with clear reasons.
04 Build foundation Vault, BRAIN_INDEX, VaultBus, progress JSON, and session registry are created.
05 Test the loop Prompts, save-up, handoff, lint, security sweep, and eval record are checked.

Lane Responsibilities

The product promise is not one tool. It is a setup co-pilot that routes the user to the right lane and leaves inspectable artifacts.

Lane What the clean-room run tests Evidence Risk to watch
Cowork Can a chat-first user mount files, answer the diagnostic, create agents, and run the handoff loop without terminal comfort? Vault folders, Chief of Staff prompt, specialist prompt, progress JSON, handoff note. Do not make Cowork feel required for every user.
Claude Code Can a terminal-native user inspect repo state, create files, run lint, understand git output, and verify the setup? Command log summary, lint report, generated prompts, QC run note. Do not assume the user already speaks git.
Codex Can an OpenAI local agent create scoped repo changes, validate them, and produce a clean handoff? Diff, local checks, browser or HTML review when relevant, handoff summary. Do not assume Claude-specific tools exist.
Blended Can Claude and Codex share one vault, divide ownership, and avoid editing conflicts? Ownership map, status files, one cross-agent directive, acknowledgements, no file conflicts. Do not route beginners here unless the diagnostic says the complexity is useful.

Quest Tracking

Gamification is the visible layer over the setup work. The repo should track progress in one durable file and optionally mirror it into a friendlier dashboard.

The source of truth is onboarding-progress.json. The co-pilot creates a local HTML dashboard by default. If Notion exists, the co-pilot can also mirror quests into a Notion coordination hub.

Gap Checks

These are the questions the clean-room run should answer before the repo is treated as ready for public users.

Check What success looks like Patch if it fails
Basic user path User understands they can start with a simple AI workflow, not only a full agent team. Add clearer website and repo positioning for basic, team, and advanced paths.
Dashboard choice User knows progress can live in JSON, Notion, or a local HTML artifact. Patch START_HERE.md Q0 and Q3 instructions.
Notion expectation User knows the repo can create a Notion dashboard, but not the maintainer's private Command Center. Patch docs/NOTION_DASHBOARD_SCHEMA.md and setup copy.
Recursive eval One breakdown becomes a lesson, doc patch, eval, record, and retest. Patch docs/EVALS.md, docs/IMPLEMENTATION_QC.md, or the relevant lane doc.
Security boundary No secrets, private routes, raw inboxes, chat exports, or production internals appear. Stop release, remove the artifact, rerun security review.
Marketing claim match Website and video only show behaviors the repo can actually guide. Change the claim, or add the missing setup support before launch.
