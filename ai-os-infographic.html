Your Toolkit

What Each Tool Does (Plain English)

Seven tools, each with a clear job. No overlap, no ambiguity.

Wi

Wispr

Voice Interface

Voice-to-text that lets you talk to your agents instead of typing. This is what makes agent management practical — you speak your directions, Wispr types them. The first thing the co-pilot sets up.

O

Obsidian

Agent Dashboard

A folder of markdown files that agents read and write. Every agent reads the BRAIN_INDEX on boot to learn where everything is. Handoffs, status files, project notes, decision logs — all live here. This is the forever memory.

N

Notion

Human Dashboard

Your dashboard for tracking everything at a glance. Agent statuses, project progress, session logs. Agents update it automatically via MCP. You see what's happening without digging through files.

V

VaultBus

Agent Coordination

How agents talk to each other without being in the same session. Your chief of staff drops a task into an agent's inbox (a markdown file). When that agent boots, it checks the inbox and finds the task waiting. Async but reliable.

W

Optional Watcher

IDE Bridge

A conservative Python bridge for chat-first setups that cannot run local checks directly. It accepts argv-only commands and asks you to approve each one in the terminal. Claude Code and Codex usually verify work directly without it.

S

SQLite

Durable Memory

The machine-readable event log. Every VaultBus message, every status change, every command — stored in a structured database. If you ever need to audit what happened, query this. Agents can also use it for message tracking and deduplication.

F

File System

Shared Storage

One folder on your desktop that every agent can read and write. Project files, code repos, VaultBus inboxes, watcher bridge files — everything lives here. This is the physical layer that makes all coordination possible.

The Setup Journey

10 Steps from Zero to Running System

This is the sequence your setup co-pilot will walk you through. Each step builds on the last.

START_HERE.md Entry Point

You paste the contents of START_HERE.md into Cowork, Claude Code, Codex, or a blended setup lead. This boots your setup co-pilot, whose job is to get you running.

Assessment Co-pilot

The co-pilot asks you a few questions: Do you have Obsidian? Notion? An IDE? What projects are you thinking about? Based on your answers, it builds a personalized lesson plan.

While the co-pilot works on your plan, it sends you here — to this infographic — so you understand why each piece exists before you set it up.

Vault Setup Obsidian

Your co-pilot creates the Obsidian vault with the standard folder structure (10 sections). This becomes the persistent memory layer — the thing that survives across every agent session, forever. You just install Obsidian and confirm.

What you'll have: A vault with BRAIN_INDEX, handoff templates, VaultBus protocol, session registry, and project folders ready for your first agents — all built by your co-pilot.

VaultBus + SQLite Coordination

Your co-pilot sets up the inter-agent messaging system: command inboxes for each agent, status files, and the SQLite event log. This is how your chief of staff will communicate with project agents.

BRAIN_INDEX Registry

Your co-pilot populates the master index — the single file every agent reads on boot. It tells them where projects live, where their inbox is, where handoffs go, and what rules to follow.

Key insight: If it's not in BRAIN_INDEX, it doesn't exist to agents. This is why "Create + Register = Atomic" is a core rule.

Notion Dashboard Human View

Your co-pilot sets up a ring-fenced Notion workspace with agent tracker, project tracker, and session log databases — connected via MCP so agents update it automatically. This is your management view.

Local Verification Bridge

Your co-pilot chooses the safest verification path for your lane. Claude Code and Codex can usually run checks directly. Cowork can use the optional watcher only when you approve each command in the terminal.

First Agent: Chief of Staff Boot

Boot your admin agent — your chief of staff. It reads BRAIN_INDEX, checks its inbox, and starts managing your system. This is the moment the system comes alive.

Magic moment: Your chief of staff reads BRAIN_INDEX and says "I see the vault, I see my inbox, I'm ready to coordinate." You have a system.

Second Agent: Project Specialist Scale

Boot a project agent for your first real project. Your chief of staff writes its boot prompt, drops setup tasks in its inbox, and monitors its progress. Two agents, coordinating.

Third Agent + Graduation Full System

Boot a third agent. Now you have a chief of staff managing two project specialists through VaultBus. You see coordination happening in real time. Nothing gets lost.

You've graduated when you understand: how to tell your chief of staff to tell an agent to save up, how handoffs work, how VaultBus keeps agents in sync, and how to add a new agent whenever you're ready.

When an agent runs out of memory

Tell your chief of staff. It tells the agent to save up. The agent writes a handoff. The chief of staff writes the next boot prompt. You paste it into a new session. The cycle continues — no context lost.

When you want a new project

Tell your chief of staff. It creates the project folder, registers it in BRAIN_INDEX, writes a boot prompt for the new project agent, and drops setup instructions in its inbox. You boot it.

Why It Doesn't Break

The Rules That Keep It Clean

Without governance, multi-agent systems degrade. These rules prevent that — each one exists because something went wrong without it.

01

Context Ceiling

Agents track their tool call count. Warning at 60, hard stop at 80. They write handoffs before they degrade — never after.

02

Create + Register = Atomic

Any new file must be registered in BRAIN_INDEX in the same action. If it's not indexed, it doesn't exist to the next agent.

03

Zero Mechanical Work

You make decisions. Agents handle everything else — file management, deployments, coordination, monitoring.

04

Versioned Handoffs

When an agent finishes, it writes a structured handoff. The next version boots with full context. Knowledge compounds.

05

Vault Lint

Automated checks catch orphaned files, broken references, and contradictions. The knowledge base stays clean as it grows.

06

Approved Local Execution

Local commands are verified through the active lane. Cowork can use an approval-required watcher. Claude Code and Codex can run local checks directly when the user approves the work.
