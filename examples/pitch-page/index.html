The Problem

AI today is a brilliant amnesiac

Most people use AI the same way: open a chat, ask a question, get an answer, close the window. Tomorrow you start over. The AI doesn't remember what you told it yesterday. It doesn't know what your other conversations were about. It can't talk to itself. You end up being the one holding all the context, doing all the coordination, repeating yourself constantly.

You're getting help from the smartest intern in the world — but an intern who shows up every morning with no memory of yesterday, no idea what their coworkers are working on, and no way to hand off their notes to the next shift.

What if AI could be a team instead of a chatbot?

Under the Hood

How the system actually works

Plain English first. Click any "Technical details" section if you want to see exactly how it's built.

1
Every agent starts by reading the master index

There's a single document that tells every agent where everything is: which projects exist, who's working on what, where to find past work, and what rules to follow. When an agent starts a new session, it reads this index first. It never asks you for context it can find itself.

Technical details →
The master index is BRAIN_INDEX.md, a structured Markdown file in an Obsidian vault. It contains project locations, agent session registry, file paths to every key document, and 13 codified governance rules. Agents query specific files on demand rather than loading the entire vault.
2
Agents coordinate through a structured message system

The admin agent sends instructions to other agents through a messaging system. Each agent has an inbox. Messages are tracked from creation through delivery to completion. If something gets stuck, the system flags it. There's a complete audit trail of every message ever sent.

Technical details →
The coordination protocol is called VaultBus. It has five layers: Status (each agent's current state), Commands (directives from admin to agents), Escalations (blockers needing attention), Events (append-only audit log), and Views (human-readable dashboards). Commands are immutable after creation. Messages follow a lifecycle state machine.
3
Context survives forever through structured handoffs

AI conversations have a limit on how much they can hold in memory. When an agent approaches that limit, it writes a detailed handoff document — what it accomplished, what's still pending, what decisions were made, and exact next steps. The next version of that agent starts by reading the handoff and picks up exactly where the last one left off.

Technical details →
This is the Context Ceiling Protocol. Agents use the clearest context signal their lane exposes. If a token or context meter exists, yellow starts around 50 percent and red starts at 70 percent. If no meter exists, tool calls are a fallback. At red, the agent checkpoints the current task, updates VaultBus, writes a structured handoff, and updates the session README. The next version starts by reading that handoff.
4
Agents can take real actions on your computer

AI agents typically run in a sandbox — they can read and write files but can't actually do things like push code to GitHub or run a build process. A secure bridge lets agents send specific, pre-approved commands to your actual machine — all without you touching anything.

Technical details →
The optional bridge is called the Watcher — a Python script running in a terminal on the host machine. It polls a JSON inbox file for argv-only commands, asks the user to approve each command, executes only allowlisted commands, and writes redacted results to an outbox file.
5
Quality is enforced through 13 governance rules

Every rule in the system exists because something went wrong without it. These aren't theoretical best practices. They're scar tissue turned into policy. As you run your own system, you'll develop your own rules alongside the starter set.

Technical details →
The 13 rules are codified in BRAIN_INDEX.md and enforced through boot prompts, handoff docs, and admin monitoring. Key examples: Rule 6 — Create + Register is atomic (any new file must be registered in BRAIN_INDEX in the same action). Rule 10 — Context Ceiling (use the real context meter when available; use tool calls only as a fallback). Rule 12 — Use Edit for existing files, not full rewrites (prevents invisible bugs). There's also a Vault Lint system that periodically scans the knowledge base for structural issues.
Governance

Six Key Rules

Part of a 13-rule framework. Each rule exists because something went wrong without it.

01

Context Ceiling

Agents watch the best context signal available. Real meters win. Tool calls are only a fallback. Agents write handoffs before they degrade, so no context is lost.

02

Create + Register = Atomic

Any new file must be registered in BRAIN_INDEX in the same action. If it's not indexed, it doesn't exist to the next agent.

03

Zero Mechanical Work

You make decisions. Everything operational — file management, deployments, coordination, monitoring — is the agents' problem.

04

Versioned Handoffs

When an agent hits its ceiling, it writes a structured handoff. The next version boots with full context. Knowledge compounds, never resets.

05

Vault Lint

Automated quality checks catch orphaned files, broken references, and contradictions. The knowledge base stays clean as it grows.

06

Approved Local Execution

Local commands are verified through the active lane. Cowork can use an approval-required watcher. Claude Code and Codex can run checks directly.

Security & Trust

Hardened by Design

The system is built with real security constraints, not afterthoughts.

Approval-Required Watcher

The public starter watcher accepts argv-only commands, uses a small allowlist, blocks dangerous patterns, and requires terminal approval before execution.

Sandboxed Agents

Each agent runs in an isolated environment. Can read/write shared files but cannot access the network, other sessions, or system resources directly.

Immutable Commands

VaultBus command files are immutable after creation (except status field). Full audit trail. No agent can edit another agent's status file.

Credential Isolation

API keys, tokens, and sensitive data stay in environment variables or Keychain. Never embed values in boot prompts, vault notes, or shared agent files.

Why This Matters

This isn't what you've seen before

Every week there's a new "AI agent" demo. Most of them book flights, send emails, and break after three steps. This is something fundamentally different.

Not Automation

Creative reasoning, not task execution

These agents don't fill out forms. They make design decisions, synthesize information, and exercise judgment. This is reasoning work, not button-pressing.

Not Ephemeral

Accumulated context over time

Most AI conversations last one session. These agents inherit context from every previous version. Nothing is ever lost. The system gets smarter over time.

Not Fragile

Governance that prevents degradation

AI agents degrade when they run too long. The Context Ceiling Protocol is enforced as law. Every failure encountered becomes a rule. The system hardens itself.

Not Narrow

Multiple domains, one system

The coordination patterns are domain-agnostic. If you can describe the work, you can build an agent for it.

Not Theoretical

Built to run in production

This framework has been used in real production across multiple projects. It's battle-tested infrastructure, not a proof of concept.

Not Dependent

Built on standard tools, no proprietary lock-in

Obsidian, Python, GitHub. No custom cloud infrastructure. No expensive APIs. The entire system runs on a desktop with a mounted folder. Portable, reproducible, and affordable.
